## 第一页

在大多数语言里，当你使用抽象时，你会付出代价。
更高级、就意味着更慢。

但 C++ 是个例外。
在这门语言中，你写的抽象代码，最终可能被编译得跟手写 C、甚至汇编一样快。

这就是 C++ 核心设计哲学：
零成本抽象（Zero‑overhead abstraction）。

今天，我们从源头解释它是什么、为什么重要，以及它如何塑造了现代 C++。

---

## 第二页
什么是抽象？这里的抽象是指中高级编程语言为了帮助人类构建程序而创造出来的高层次概念。C++ 聚焦的领域（OS、数据库、游戏引擎、量化、高性能计算）需要构建巨大而复杂的软件系统。
没有抽象，人类无法驾驭如此复杂的代码。我们要开发大型的复杂的软件，不可能直接用汇编或者机器码开发，所以必须要借助抽象，例如：函数（function）、结构体（struct）、面向对象（class, OOP）、模块（module / 包）、泛型、模板（template）、面向方面（AOP）。这些概念都是程序语言设计者为了让你“更容易构建大型程序”而引入的概念，它们本质上并不属于CPU和硬件指令集，在机器语言里本不存在。

## 第三页

C++的设计哲学是努力保证这些高级概念（也就是所说的“抽象”）并不会让程序变慢，这就是所谓的“零成本”的含义。虽然我不敢说C++完全达到了“零抽象成本”的目标，但是在C++的演化过程中、在每次决策中，这一条始终是被优先考虑的。

具体而言，希望达到这样的效果：

当用C++这些抽象写代码时，性能不比不用抽象更慢；不会额外引入运行时开销；如果不用抽象，就不会产生任何额外代价。
如果你用 C 语言手动实现一个带有多态特性的结构（例如通过函数指针模拟虚函数），它的性能不会比 C++ 原生的虚函数机制更高。C++ 的多态实现已经是“你自己实现也不会更快”的程度。
而如果在 C++ 中你不使用类的多态特性（也就是不定义虚函数），那么类的对象布局就不会包含虚表指针，也就没有任何额外的运行时开销。此时这个类的成本与 C 语言中的结构体 struct 完全相同。

## 第四页

下面是 C++ 如何实现“零成本抽象” 的方法的总结：

零成本抽象的实现方法总结：
1. **编译期计算 / 展开（模板 + 静态多态）**
2. **按需付费：特性不使用就没有任何成本**
3. **与硬件直接映射（低层抽象不会遮挡机器模型）**
4. **内联与优化接口（inline / likely / noexcept 等）**
5. **避免全局虚拟机器（没有统一的运行时）**

## 第五页

编译期模板推导展开以及编译期计算。

- 模板推导和展开
- 静态多态（CRTP等）
- 编译期求值（constexpr、consteval、fold expression）

C++ 的模板在编译期实例化，模板函数会被展开、内联，”模板“这一抽象在编译后就被“消失”了。
一般的多态是在运行时才会决定实际执行的是哪个函数，但是静态多态（CRTP 等）在编译期就确定了实际的执行的函数。因此静态多态没有一般的多态所需要的代价。

编译期求值在编译期间完成计算，运行时直接使用计算结果。下面是一个例子：

```cpp

template <typename... Ts>
constexpr double l2_norm(Ts... xs) {
    // fold: (x*x + ...)
    double sum = (... + (xs * xs));
    return std::sqrt(sum);
}

constexpr double r = l2_norm(3.0, 4.0); // r = 5.0 (编译期完成)

```

因为在编译期完成了工作，所以运行时的开销几乎为零。


## 第六页
按需付费：特性不使用就没有任何成本

- 不用异常，就没有异常开销  
- 不用 RTTI，就没有 RTTI
- 不用多态，就没有虚表  
结果：**不用，就等于成本不存在。**

这句话的意思非常直接：

**如果你在 C++ 代码里没有使用 RTTI（运行时类型识别），那么编译器就不会为你的程序生成任何 RTTI 相关的数据结构，也不会带来任何运行时开销。**

换句话说：

- 你没用 `dynamic_cast`
- 也没用 `typeid`
- 也没开启需要 RTTI 的特性

那么：

- 不会生成类型信息表
- 不会有类型名称字符串
- 不会增加额外的对象元数据
- 不会占用运行时内存
- 不会有额外的检查行为

**等同于 RTTI 在你的程序里“完全不存在”。**

这就是 “不用抽象，就没有成本” 的典型例子。

## 第七页

与硬件直接映射
C++ 的底层特性紧贴真实硬件：对象布局、内存访问、指针行为都与机器模型一致。语言不会插入 VM、GC 这类中间层，不会遮挡硬件细节，使你能直接控制内存、布局和性能。

这对“零成本抽象”很重要，因为抽象不会强制增加运行时负担，也不会妨碍你预测和优化实际的机器行为。

一句话总结：C++ 的抽象不遮挡硬件，让代码行为可控、可预测、可优化。
C++ 的设计哲学是：  
> 抽象能帮你写得更容易，但不会妨碍你直接控制硬件。

不像某些语言（例如 Java 或 Python）：

- 所有对象都必须堆上分配  
- 内存由 GC 管理  
- 类型布局由 VM 决定  
- 运行时模型与你写的代码不一致  

这些都是“抽象把硬件细节遮挡掉”的例子。

而 C++ 刻意避免这样做。

### 为什么这对“零成本抽象”很重要？  
因为如果语言的抽象层太厚：

- 你无法准确控制性能  
- 你无法预测真正的对象布局  
- 你无法决定何时分配 / 释放资源  
- 你无法消除不必要的隐藏成本  

## 第八页

无虚拟机、无垃圾回收，编译期优化手段

C++ 不依赖虚拟机、统一对象模型或垃圾回收器，也没有全局运行时调度器。语言层没有强制性的中间层，因此不会为抽象引入固有成本。

编译器可以基于语言的静态特性进行激进优化，包括：  
- 内联函数，消除调用开销  
- `noexcept` 帮助移除异常路径带来的额外检查  
- `[[likely]]`、`[[unlikely]]` 改善分支预测  
- 利用类型信息进行更高效的数据布局和优化

这些机制共同保证：  
C++ 提供的抽象几乎不带运行时成本，不使用的特性不会产生任何额外代价。
